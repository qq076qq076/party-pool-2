# 網頁派對遊戲規格書（MVP v0.2）

## 1. 專案概述

### 1.1 專案名稱（暫定）
- `Party Pool：試膽派對`

### 1.2 專案目標
- 製作一款「主畫面 + 手機控制器」的多人網頁派對遊戲。
- 遊戲節奏與體驗參考《雞膽2─試膽挑戰─》（TORIDAMA2）類型：短回合、緊張感、派對互動。
- 玩家可透過掃描 QR Code 或輸入房間碼快速加入。

### 1.3 目標平台
- 主畫面（Host/Display）：桌機瀏覽器（Chrome/Edge）。
- 控制器（Player Controller）：手機瀏覽器（iOS Safari / Android Chrome）。

---

## 2. 產品定位與範圍

### 2.1 核心定位
- 同地同樂（Couch Party）多人互動遊戲。
- 主畫面負責顯示遊戲內容；每位玩家以手機作為專屬控制器。
- 整體主題：`生存力測試`。

### 2.2 本階段範圍（MVP）
- 建立房間
- 加入房間（QR + 房間碼）
- 房間等待室（顯示人數、玩家清單、連線狀態）
- 反應型 mini game（第一關）
- 回合結束與結果畫面

### 2.3 暫不納入（Post-MVP）
- 帳號系統/好友系統
- 語音聊天
- 排行榜/賽季制
- 複雜配對（跨區自動配房）

### 2.4 美術方向（已定案）
- 整體風格：可愛 + 搞怪的日式派對風格
- 呈現目標：角色與介面辨識度高、表情誇張、色彩明亮

---

## 3. 使用者角色

- `房主（Host）`
  - 建立房間、顯示 QR 與房間碼、進入回合準備階段。
- `玩家（Player）`
  - 透過手機加入房間、在每回合作為控制器輸入操作。

---

## 4. 核心使用流程

### 4.1 主流程
1. 進入首頁，看到 `開房間` 與 `加入遊戲`。
2. 房主點 `開房間`：
   - 系統建立房間並生成房間碼（例：`ABCD`）。
   - 顯示房間內畫面：房間碼、QR Code、目前人數。
3. 玩家以手機：
   - 掃 QR Code 直接進入加入頁（帶入房間碼），或
   - 手動輸入房間碼加入。
4. 玩家輸入暱稱後即可進房（MVP 不以 DeviceMotion 授權作為入房阻擋），主畫面即時更新人數與清單。
5. 房主按 `開始準備` 後進入「回合準備階段」，伺服器啟動 60 秒準備倒數（不會立刻開局）。
6. 顯示玩法說明與 `OK` 按鈕，玩家可在倒數內完成準備。
7. 若全員在 60 秒內按 `OK`，伺服器立即開局；若有人未按 `OK` 或斷線，60 秒到仍由伺服器自動開始。
8. 房主不能手動強制開局；房主僅能觸發「進入準備階段」。
9. 回合開始後玩家用手機操作，主畫面顯示結果。
10. 顯示最終結果，可選 `再來一局` 或 `返回大廳`。

### 4.2 例外流程
- 房間碼錯誤：提示「房間不存在」。
- 房間已滿：提示「房間已滿」。
- 遊戲已開始：提示「本局進行中，可排隊下一場（可選）」。
- 準備階段玩家斷線：倒數不中止，等 60 秒結束自動開局。
- 回合進行中玩家斷線：該回合不補時，下一回合可重連。

## 5. 功能需求

### 5.1 首頁
- 按鈕：`開房間`、`加入遊戲`
- 簡短說明：手機掃碼加入、建議開啟音效

### 5.2 開房間
- 建立唯一房間 ID 與短房間碼
- 顯示：
  - 房間碼（大字）
  - QR Code（可掃描）
  - 目前人數/最大人數（例：`3 / 8`）
  - 玩家名單（含連線狀態）
- 房主操作：
  - 開始準備（進入準備階段）
  - 關閉房間（可選）

### 5.3 加入遊戲（手機）
- 方式 A：掃碼（URL 含 room_code）
- 方式 B：手動輸入房間碼
- 輸入暱稱
- MVP 入房必要條件：暱稱有效 + 房間可加入（不以 DeviceMotion 授權作為入房門檻）
- DeviceMotion 權限策略（為支援 `shake` 關卡）：
  - 進入需要 `shake` 的回合前再要求授權，或由玩家在等待頁先行授權
  - 若拒絕或裝置不支援：仍可留在房內；需要 `shake` 的回合顯示不可操作提示並以該回合 0 分計算
- 權限被拒時需提供原因與重新開啟引導

### 5.4 房間管理
- 房間碼格式：4~6 位英數大寫（MVP 建議 4 位）
- 每房最大人數：8（MVP 可調為 4~8）
- 房間生命周期：
  - 建立後 10 分鐘未開始自動釋放
  - 結束後 5 分鐘保留可重開

### 5.5 手機控制器（依關卡動態切換）
- 本規格先定義「輸入能力」，不綁定固定按鍵版型。
- 每個關卡於開始時下發該關卡的 `control_profile`，控制器 UI 依設定切換。
- MVP 輸入能力範圍：
  - 點按（Tap）/ 連打（Mash）/ 長按（Hold）
  - 滑動（Swipe）
  - 搖晃（Shake，使用裝置動態感測）
- 狀態顯示：
  - 連線中 / 已連線 / 斷線重連
  - 當前回合操作提示（依關卡顯示）
  - 感測器狀態（可用 / 未授權 / 不支援）
  - 回合可操作狀態（可操作 / 本回合不可操作）
- 若 `control_profile.mode = shake` 且感測器狀態為未授權或不支援，控制器需顯示「本回合不可操作」，並停止送出 `player_input` 事件。

### 5.6 遊戲回合（MVP：第一關）
- 主題：生存力測試
- 第一個 mini game：固定時間連打（Tap Challenge）
- 規則：在固定時間內，點擊次數最多者為該關第一名
- 回合長度：20 秒（可配置）
- 回合流程：
  - 玩法說明畫面（顯示規則、操作方式、目標）
  - 玩家準備確認（每位玩家手機按 `OK`）
  - 準備階段最長 60 秒（全員 `OK` 立即開始；逾時自動開始）
  - 下發本回合控制設定（`control_profile`）
  - 倒數開始（3 秒）
  - 進行中（主畫面呈現、手機輸入）
  - 結算（排名與加分）
- 計分規則：
  - 每關只給第一名 +1 分
  - 若第一名同分，並列第一者都 +1 分
- 整局排名規則：
  - 以 3 回合總分排序
  - 總分同分則並列名次（MVP 不加賽）
- 結束後操作規則：
  - `再來一局`：保留房間與玩家清單，重置分數、回合序號、ready 狀態，並從第 1 回合重新開始
  - `返回大廳`：返回等待室並保留房間碼，房主可再次進入準備階段
- 局數：3 回合（可配置）

### 5.7 語言支援
- 支援語言：English / 繁體中文
- 預設語言：依瀏覽器語言自動判斷
  - `zh-TW` 或 `zh-*` -> 繁體中文
  - 其他 -> English
- 玩家可在首頁與手機控制器頁手動切換語言
- 系統記住同裝置上次語言選擇（local storage）

### 5.8 重連與身份綁定（MVP 必做）
- 玩家首次入房後由伺服器發放 `rejoin_token`（僅本房間有效，綁定 `player_id`）
- 手機端將 `rejoin_token` 暫存 local storage
- 斷線後重開頁面可用 `room_code + rejoin_token` 自動找回原玩家席位
- 若 token 驗證失敗或席位已被占用，回傳錯誤並要求手動重新加入
- 準備階段斷線不影響開局時間：仍以 60 秒規則自動開局
- 回合進行中斷線：該玩家該回合停止累計，下一回合可恢復

## 6. 非功能需求

### 6.1 效能
- 控制輸入到主畫面反應延遲：目標 < 120ms（同區網路）
- 主畫面 FPS：目標 60 FPS（最低可接受 30 FPS）
- 支援同房 8 人穩定連線

### 6.2 相容性
- 主畫面：Chrome/Edge 最新兩個大版本
- 手機：iOS Safari、Android Chrome 最新兩個大版本
- 直橫向：
  - 主畫面固定橫向
  - 手機控制器建議直向（可鎖）

### 6.3 可用性
- 掃碼到加入成功步驟 <= 3 步
- 房間碼需高對比、可遠距辨識
- 斷線需可自動重連（同裝置同瀏覽器）

## 7. 技術架構（建議）

### 7.1 前後端組成
- `Web 客戶端（主畫面）`
  - 遊戲畫面、房間顯示、回合控制
- `Mobile Web 控制器頁（HTML5）`
  - 玩家輸入 UI、連線狀態、回合提示
- `即時連線伺服器（WebSocket）`
  - 房間管理、玩家連線、事件轉發

### 7.2 連線協定（事件制）

#### 7.2.1 訊息封包格式（雙向統一）
- `event`: string（事件名稱）
- `request_id`: string（Client 送出事件必填，建議 UUID）
- `sent_at`: number（Unix timestamp，毫秒）
- `payload`: object（事件資料）

#### 7.2.2 Client -> Server（MVP）
- `create_room`
  - payload: `{ "max_players": number, "locale": "zh-TW" | "en" }`
- `join_room`
  - payload: `{ "room_code": string, "nickname": string, "client_capabilities": { "motion_supported": boolean }, "rejoin_token"?: string }`
- `host_enter_ready_phase`（原 `start_game`）
  - payload: `{ "room_code": string }`
- `player_ready_ok`
  - payload: `{ "room_code": string, "round_no": number }`
- `player_input`
  - payload: `{ "room_code": string, "round_no": number, "input_type": "tap" | "swipe" | "shake", "input_value": number, "ts_client_ms": number }`
- `motion_permission_result`
  - payload: `{ "room_code": string, "status": "granted" | "denied" | "unsupported" }`
- `request_rejoin`
  - payload: `{ "room_code": string, "rejoin_token": string }`
- `heartbeat`
  - payload: `{ "room_code": string, "ts_client_ms": number }`

#### 7.2.3 Server -> Client（MVP）
- `room_created`
  - payload: `{ "room_id": string, "room_code": string, "max_players": number }`
- `room_state_updated`
  - payload: `{ "room_code": string, "status": "waiting" | "readying" | "playing" | "ended", "players": Player[] }`
- `ready_timer_started`
  - payload: `{ "room_code": string, "round_no": number, "ready_deadline_at": number }`
- `ready_status_updated`
  - payload: `{ "room_code": string, "round_no": number, "ready_count": number, "total_count": number }`
- `ready_timeout_reached`
  - payload: `{ "room_code": string, "round_no": number }`
- `game_started`
  - payload: `{ "room_code": string, "round_no": number, "started_at": number }`
- `control_profile_updated`
  - payload: `{ "room_code": string, "round_no": number, "control_profile": ControlProfile }`
- `round_started`
  - payload: `{ "room_code": string, "round_no": number, "start_at": number, "duration_sec": number }`
- `round_result`
  - payload: `{ "room_code": string, "round_no": number, "ranking": RoundRanking[], "scoreboard": ScoreboardItem[] }`
- `match_result`
  - payload: `{ "room_code": string, "final_ranking": FinalRankingItem[], "winners": string[] }`
- `error`
  - payload: `{ "request_id"?: string, "code": string, "message_key": string }`

#### 7.2.4 `error.code`（MVP）
- `ROOM_NOT_FOUND`
- `ROOM_FULL`
- `ROOM_NOT_JOINABLE`
- `INVALID_ROOM_CODE`
- `INVALID_NICKNAME`
- `DUPLICATE_NICKNAME`
- `REJOIN_TOKEN_INVALID`
- `REJOIN_SLOT_TAKEN`
- `NOT_ROOM_HOST`
- `ROUND_NOT_READY`

### 7.3 狀態同步策略
- 伺服器權威（Server Authoritative）：
  - 房間狀態、回合開始/結束、分數由伺服器決定
- 客戶端僅負責輸入與渲染，避免作弊與狀態分歧

### 7.4 通訊與部署安全
- 正式環境必須使用 `HTTPS`
- 即時連線必須使用 `WSS`（WebSocket Secure）
- 不允許混合內容（HTTPS 頁面呼叫 WS）

## 8. 資料模型（MVP）

### 8.1 Room
- `room_id`: string（UUID）
- `room_code`: string（短碼）
- `host_socket_id`: string
- `status`: `waiting | readying | playing | ended`
- `max_players`: number
- `players`: Player[]
- `ready_deadline_at`: timestamp
- `created_at`: timestamp

### 8.2 Player
- `player_id`: string
- `nickname`: string
- `socket_id`: string
- `is_connected`: boolean
- `ready_status`: `pending | ok`
- `rejoin_token_hash`: string
- `score`: number
- `last_seen_at`: timestamp

## 9. UI 畫面清單

1. 首頁（開房間 / 加入遊戲）
2. 開房後房間頁（房間碼 + QR + 玩家名單 + 人數）
3. 手機加入頁（房間碼輸入/帶入 + 暱稱 + 權限同意）
4. 手機等待頁（連線成功、等待開始）
5. 回合說明 + OK 準備頁（含 60 秒倒數）
6. 遊戲回合頁（主畫面）
7. 結算頁（單回合/最終）

---

## 10. 安全與風險

- 房間碼防爆破：
  - 同 IP 短時間輸入失敗次數限制
- 基本防濫用：
  - 暱稱長度限制與敏感字過濾（可後補）
- 連線風險：
  - 行動網路切換造成斷線，需要重連邏輯
- 權限風險：
  - 使用者拒絕動態感測權限會導致 `shake` 類回合無法操作（但不阻擋入房）

---

## 11. 驗收標準（MVP Done）

- 可由主畫面成功建立房間並看到房間碼與 QR Code
- 至少 2~8 位玩家可透過手機加入同一房間
- 玩家可不授權 DeviceMotion 仍加入房間；若回合需要 `shake`，未授權或不支援者會被標記為該回合不可操作
- OK 準備最長 60 秒，逾時可自動開始回合
- 房主無法手動強制開始回合
- 玩家手機輸入可即時控制主畫面互動
- 可完整跑完一局（至少 3 回合）並顯示結果
- 每關第一名 +1，同分第一皆 +1
- 斷線後可使用 token 重連回同房間

- 開局控制：房主僅能觸發準備階段，正式開局由伺服器依 OK/逾時條件決定
- 美術風格：可愛 + 搞怪日式風格
- 權限規則：DeviceMotion 非入房門檻，僅在 `shake` 關卡需要
- 重連規則：`rejoin_token` 為 MVP 必做
- 語言：English 與繁體中文可切換，預設依瀏覽器語言
- 通訊與部署：正式環境使用 HTTPS + WSS
