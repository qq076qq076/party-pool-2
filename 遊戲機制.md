# 遊戲機制設計（MVP v0.1）

## 1. 核心主題
- 主題：`生存力測試`
- 設計目標：每一關都在測試玩家「反應力、專注力、操作穩定性」。

---

## 2. `control_profile` 白話說明

`control_profile` 可以把它想成「這一關手機要怎麼操作的說明卡」。

- 每一關開始前，伺服器會送一份 `control_profile` 到每支手機。
- 手機看到這份資料後，會自動切成該關需要的操作 UI（例如連打按鈕、搖晃提示、滑動區）。
- 好處是：
  - 不用把控制器寫死成固定按鍵。
  - 新增新關卡時只要換設定，不用重做整個控制器頁。

### 2.1 `control_profile` 最小欄位（MVP）
- `mode`：關卡操作模式（`tap` / `swipe` / `shake`）
- `instruction_key`：對應語系文案 key（例如 `round1.tap_fast`）
- `round_duration_sec`：回合長度（秒）
- `ready_timeout_sec`：準備階段最長等待秒數（本專案固定 60）
- `countdown_sec`：正式開始前倒數秒數（預設 3）
- `allow_input_before_start`：倒數前是否可提前輸入（MVP 為 `false`）

### 2.2 第一關範例
```json
{
  "mode": "tap",
  "instruction_key": "round1.tap_fast",
  "round_duration_sec": 20,
  "ready_timeout_sec": 60,
  "countdown_sec": 3,
  "allow_input_before_start": false
}
```

## 3. 回合前準備規則（OK 機制）
- 進入每回合時，先顯示玩法說明。
- 每位玩家在手機上按 `OK` 表示已準備。
- 準備階段最長 60 秒。
- 房主只能按「開始準備」觸發倒數，不能直接強制開局。
- 若全員提前按 `OK`，立即開局。
- 若 60 秒到仍有人未按 `OK` 或斷線，系統仍自動開局。
- 房主不能強制開始。

---

## 4. 第一個 mini game：極限連打（Tap Challenge）

### 4.1 玩法
- 在固定時間內（20 秒）盡可能快速點擊手機上的按鈕。
- 點擊次數即該關成績。

### 4.2 勝負與計分
- 該關最高點擊數者為第一名。
- 每關只有第一名得分。
- 第一名得分：`+1`。
- 若第一名同分，並列第一的玩家都 `+1`。
- 整局（3 回合）以總分排行；總分同分時並列名次，MVP 不加賽。
- 玩家選 `再來一局` 時保留房間與玩家名單，重置分數與回合數重新開始。

### 4.3 反作弊（MVP）
- 伺服器以事件時間戳與速率上限做基本過濾。
- 異常高頻輸入可標記並忽略超出上限的事件。

---

## 5. 權限規則
- MVP 入房不強制 DeviceMotion；玩家輸入暱稱後即可加入房間。
- DeviceMotion 只在 `shake` 關卡需要，可在等待頁預先授權或在該回合前授權。
- 若拒絕或裝置不支援：可留在房內，但 `shake` 回合顯示「本回合不可操作」並以 0 分計算。
- 權限被拒時顯示原因與引導（如何重新開啟權限）。

---

## 6. 重連策略（MVP 必做）
- 玩家入房時取得 `rejoin_token`，存在 local storage。
- 重新開啟頁面時，帶 `room_code + rejoin_token` 嘗試回到原座位。
- 若 token 無效或席位已被占用，需提示玩家手動重新加入。
- 準備階段斷線：不延長 60 秒倒數。
- 回合進行中斷線：該回合不補時，下一回合可恢復。
